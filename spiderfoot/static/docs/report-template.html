Detailed Findings
High Missing DMARC Record Email Security
What we found No DMARC DNS record exists for example.com.
Evidence dig TXT _dmarc.example.com → NXDOMAIN
Risk Attackers can send emails that appear to come from @example.com, enabling phishing attacks against your customers or partners.
How to Fix

Add a DMARC TXT record to your DNS: _dmarc.example.com IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com". Start with p=none to monitor, then move to p=quarantine or p=reject once confirmed. Takes 5–10 minutes via your DNS provider.

High Server Version Disclosed in HTTP Headers Web Infrastructure
What we found The web server returns its software name and version number in every HTTP response.
Evidence Server: Apache/2.4.51 (Ubuntu)
Risk Attackers can instantly identify known vulnerabilities for this specific server version without any manual reconnaissance.
How to Fix

For Apache: add ServerTokens Prod and ServerSignature Off to your Apache config. For Nginx: add server_tokens off; to the http block. For Cloudflare users: enable "Security Level" and hide server headers via Transform Rules.

High Unauthenticated Staging Environment Exposed Access Control
What we found staging.example.com is publicly accessible with no authentication, returning a live application.
Evidence https://staging.example.com → HTTP 200, App version 2.1.0-dev
Risk Dev environments often contain test credentials, debug interfaces, and unpatched code. Attackers commonly target staging environments to find vulnerabilities before production.
How to Fix

Restrict staging.example.com to specific IP addresses (your office/VPN) using your hosting provider's firewall or Cloudflare Access. Alternatively, add HTTP Basic Auth as a minimum barrier. If the environment is no longer needed, remove the DNS record entirely.
